### Code Reviews 
#### What and Why
- Code is a living document. There are many ways to ensure that it works for the people who use it. This is one. 
- Knowing what one is, is a differentiator from CS college grads.
- What is it? - Code review is systematic examination of computer source code. It is intended to find mistakes overlooked in the initial development phase, improving the overall **quality** of software.
- quality - bugs, best practices, elegant and simplified solutions.

**There are many types of code reviews:**

- Email thread
- Pair programming
- Over the shoulder
- Tool assisted - Pull Requests

The most common in the work place are **tool assisted** and **pair programming**.

- Github can help us: [https://help.github.com/articles/creating-a-pull-request/](https://help.github.com/articles/creating-a-pull-request/)
- Show an example of a PR on a project they are using - express: [https://github.com/expressjs/express](https://github.com/expressjs/express)

#### Github Walk through
- PR (Pull Request)
- Comment on the whole PR
- Comment on the diff of an individual file
- Comment on the whole file
- Link other people to bring them into the conversation

#### What to comment on?
- Positive and constructive critique
- Possible bugs, based on your understanding of the problem being solved. OR Good catches of bugs the developer foresaw. 
- Things that could be solved better with fewer lines of code or with a different tool or pattern. OR Things solved in a very elegant way.
- Open questions you see about the problem or the solution.
